# Роли, пользователи и контроль доступа

## Общая модель

В системе есть два разных вида учётных записей:

1. встроенный системный администратор `superadmin`, который не является сотрудником;
2. обычные пользователи из таблицы `users`, каждый из которых обязательно связан ровно с одним сотрудником через `employee_id`.

Один сотрудник не может иметь несколько учётных записей. У обычного пользователя может быть несколько ролей.

Эффективные разрешения пользователя являются объединением разрешений его ролей. Область данных вычисляется **отдельно для каждого разрешения**. Это важно при совмещении ролей: глобальное право управлять настройками не расширяет отделское право просмотра сотрудников.

Пароли не сохраняются в открытом виде. Для каждого пароля используется соль, после чего пароль хешируется через `scrypt`.

## Встроенный Superadmin

Системный администратор хранится отдельно от сотрудников в таблице `system_admin_accounts`.

Готовые данные для входа:

```text
Логин: superadmin
Пароль: superadmin
```

Особенности:

- не связан с сотрудником, отделом или предприятием;
- имеет глобальный доступ ко всем административным и кадровым модулям;
- не имеет персональной карточки сотрудника;
- его логин нельзя изменить;
- его нельзя удалить или заблокировать;
- роль `Superadmin` нельзя назначить обычному пользователю;
- при обновлении старой базы прежние назначения роли `Superadmin` сотрудникам автоматически снимаются.

## Системные роли

Системные роли создаются миграцией и защищены SQLite-триггерами. Их нельзя удалить, переименовать, изменить область или набор разрешений.

### Superadmin

- используется только встроенной системной учётной записью;
- глобальная область данных;
- полный доступ ко всем модулям, кроме персонального профиля сотрудника, поскольку системный администратор сотрудником не является.

### Сотрудник

- область `self`;
- просмотр собственной карточки;
- просмотр собственных отпусков;
- просмотр собственных начислений;
- изменение кадровых, отпускных и зарплатных данных не разрешено.

### Директор предприятия

- область `enterprise`;
- просмотр сотрудников, структуры, вакансий, кандидатов, отпусков и начислений своего предприятия;
- роль можно назначить только сотруднику, который указан генеральным директором хотя бы одного предприятия;
- после снятия назначения роль автоматически отзывается, если сотрудник больше не руководит ни одним предприятием.

### Начальник отдела

- область `department`;
- просмотр сотрудников, структуры, подбора и отпусков своего отдела;
- роль можно назначить только сотруднику, который указан директором хотя бы одного отдела;
- после снятия назначения роль автоматически отзывается, если сотрудник больше не руководит ни одним отделом.

## Кастомные роли

Superadmin может создать роль с названием, описанием, областью данных и набором разрешений.

Доступные области:

- `global` — все данные системы;
- `enterprise` — данные предприятия сотрудника;
- `department` — данные отдела сотрудника;
- `self` — только собственные данные сотрудника.

Правила:

- роль должна содержать хотя бы одно разрешение;
- `access.manage` и `settings.manage` допускаются только в глобальной области;
- роль с `access.manage` должна также содержать `employees.view`, поскольку пользователь обязательно выбирается из сотрудников;
- роль нельзя удалить, пока она назначена хотя бы одному пользователю.

## Пользователи-сотрудники

При создании обычного пользователя необходимо указать:

- сотрудника;
- уникальный логин, отличный от `superadmin`;
- временный пароль;
- статус;
- одну или несколько ролей, кроме `Superadmin`.

По умолчанию временный пароль требуется сменить при первом входе. Superadmin может сбросить пароль позднее.

При переводе сотрудника в неактивный статус связанная учётная запись автоматически блокируется.

## Запуск и вход

Экран первичной привязки администратора к сотруднику больше не используется. После применения миграций приложение сразу показывает форму входа с заполненными данными `superadmin / superadmin`.

Сессия хранится только в процессе Electron и завершается при выходе или закрытии приложения.

## Серверное применение прав

Проверки выполняются в Electron backend до вызова сервисов и репозиториев:

- проверяется наличие активной сессии;
- проверяется разрешение для операции;
- определяется область именно этого разрешения;
- списки дополняются обязательными фильтрами области;
- чтение, создание, изменение и удаление отдельных записей проверяются повторно;
- подбор, отпуска, начисления, организационная структура и dashboard также ограничиваются областью данных.

Поэтому скрытие элементов интерфейса не является единственным механизмом защиты: прямой вызов IPC за пределами разрешений или области также отклоняется.

- `010_harden_access_lifecycle.sql` — защита и отзыв устаревших руководящих ролей;
- `011_add_builtin_superadmin.sql` — отдельная системная учётная запись `superadmin`, готовый пароль и запрет назначения этой роли сотрудникам.
